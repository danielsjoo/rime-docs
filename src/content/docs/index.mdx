## Two ways to use Rime

Rime can be a no-code analysis tool, a polyglot scripting runtime, or both in the same DAG.

<CardGrid>
  <Card title="Core nodes for the common path" icon="approve-check">
    Use built-ins for the work every analysis repeats: load files, filter rows, derive columns, aggregate, join, pivot, sort, select, concatenate, and run statistical checks. The runtime handles validation, caching, previews, and report output.
  </Card>
  <Card title="Script nodes for real language work" icon="puzzle">
    When the built-ins are not enough, write a function in Python, R, JavaScript, or SQL. Name the inputs, return tables or objects, and Rime handles dataframe I/O, artifacts, stdout, figures, and downstream wiring.
  </Card>
  <Card title="One report from the whole DAG" icon="document">
    Both modes feed the same run. Rime records node status, output previews, stats, figures, and logs so the result can be shared as a single HTML narrative instead of a loose pile of files.
  </Card>
</CardGrid>

## Why Rime

<CardGrid stagger>
  <Card title="Functions, not jobs" icon="approve-check">
    A node is a function over dataframes — not a task that wires I/O. You write what each step computes; the runtime owns reading, writing, serialization, and language boundaries. The dbt mental model, extended past SQL.
  </Card>
  <Card title="One DAG, four languages" icon="puzzle">
    SQL for joins, Python for ML, R for stats, JavaScript for everything else. Same pipeline, named slots, typed boundaries. Dataframes cross language borders over Arrow IPC — zero-copy where possible, constant-time cast to pandas / tibble / DuckDB table.
  </Card>
  <Card title="Reproducible by default" icon="seti:lock">
    Content-addressed caching, deterministic outputs, freeze-able snapshots. Same script + same inputs = same artifact, every time. No "works on my machine."
  </Card>
  <Card title="Publishable narratives" icon="document">
    Render a publishable HTML report directly from your DAG. Tables, stats, stdout, figures, and node status — one render step, one document, one source of truth.
  </Card>
</CardGrid>
